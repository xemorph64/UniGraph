# UniGRAPH SQL Input Dataset Structure (Final v1)

## 1. Scope
This document defines the finalized input dataset for UniGRAPH in the Indian banking context.

This specification is source-only:
- It models how transaction and KYC data are stored in core banking systems and payment channels.
- It excludes all fraud conclusions generated by UniGRAPH.

Channels in scope: UPI, IMPS, NEFT, RTGS, CASH, SWIFT.

## 2. Design Principles
- One row per transaction event (posted, failed, reversal, chargeback, adjustment).
- Preserve raw payment references used by Indian rails (for example UTR, RRN, UPI reference, UETR).
- Keep KYC in customer/account master tables, not duplicated as derived risk outputs in transaction rows.
- Do not overwrite lifecycle history; add a new event row and link via `reference_txn_id`.

## 3. Row Definition (Which Rows)

| Row category | transaction_event_type | Meaning | Include in reporting |
|---|---|---|---|
| Financial movement | POSTED | Successfully posted financial movement | Yes |
| Failed attempt | FAILED | Attempted but failed transaction | Optional |
| Reversal | REVERSAL | Reversal of a previously posted transaction | Yes |
| Chargeback/dispute | CHARGEBACK | Network or customer dispute lifecycle event | Yes |
| Adjustment/correction | ADJUSTMENT | Manual/system correction event | Yes |

Rules:
- Keep one event per row.
- Use `reference_txn_id` for REVERSAL/CHARGEBACK/ADJUSTMENT link-back.
- Do not mutate original posted records.

## 4. Final v1 Input Schema (Strict)

### 4.1 Core Transaction Table
Table name: `transactions_input`

| Column | SQL type | Null | Key/Index | Description |
|---|---|---|---|---|
| txn_id | VARCHAR(64) | No | Primary Key | Unique transaction event identifier |
| transaction_event_type | VARCHAR(20) | No | Index | POSTED, FAILED, REVERSAL, CHARGEBACK, ADJUSTMENT |
| reference_txn_id | VARCHAR(64) | Yes | Index | Parent transaction id for lifecycle links |
| from_account | VARCHAR(32) | No | Index | Debited account identifier |
| to_account | VARCHAR(32) | Yes | Index | Credited account identifier |
| customer_id | VARCHAR(32) | No | Index | Initiating customer id/CIF link |
| amount | DECIMAL(18,2) | No | Index | Transaction amount |
| currency | CHAR(3) | No |  | ISO currency code (default INR) |
| channel | VARCHAR(16) | No | Index | UPI, IMPS, NEFT, RTGS, CASH, SWIFT |
| txn_status | VARCHAR(20) | No | Index | SUCCESS, FAILED, PENDING, REVERSED, CHARGEBACK |
| txn_timestamp | TIMESTAMP | No | Index | Transaction event time |
| value_date | DATE | Yes | Index | Settlement/value date |
| source_system | VARCHAR(40) | No |  | Origin system (for example Finacle/CBS channel source) |
| branch_code | VARCHAR(16) | Yes | Index | Branch/SOL code |
| ifsc_code | CHAR(11) | Yes | Index | Origin IFSC where applicable |
| external_ref_type | VARCHAR(20) | Yes | Index | UTR, RRN, UPI_REF, UETR, OTHER |
| external_ref_no | VARCHAR(64) | Yes | Index | External network reference number |
| narration | VARCHAR(255) | Yes |  | Raw transaction narration |
| device_fingerprint | VARCHAR(128) | Yes |  | Device identity hash/token |
| ip_address_token | VARCHAR(128) | Yes |  | Tokenized or hashed IP |
| geo_lat | DECIMAL(9,6) | Yes |  | Latitude if available |
| geo_lon | DECIMAL(9,6) | Yes |  | Longitude if available |
| maker_user_id | VARCHAR(64) | Yes |  | Maker identifier (if maker-checker flow applies) |
| checker_user_id | VARCHAR(64) | Yes |  | Checker/authorizer identifier |
| auth_timestamp | TIMESTAMP | Yes |  | Authorization timestamp |
| created_at | TIMESTAMP | No |  | Source record creation timestamp |
| updated_at | TIMESTAMP | Yes |  | Source record last update timestamp |

## 5. Forbidden Columns in Input Dataset
The following must not appear in `transactions_input`, `accounts_input`, `customers_kyc_input`, or `transaction_channel_details`:

- `risk_score`
- `ml_score`
- `risk_level`
- `recommendation`
- `is_flagged`
- `alert_id`
- `rule_violations_json`
- `primary_fraud_type`
- `fraud_type`
- `is_fraud`
- `shap_top3`
- `ensemble_score`, `drools_score`, `gnn_score`, `isolation_score`
- `pmla_citation`, `fatf_typology`

## 6. Where Derived Data Should Live
Derived outputs generated by UniGRAPH must be stored in downstream tables/topics, for example:
- `transaction_enriched`
- `transaction_risk_assessment`
- `alerts`
- `cases`
- `reports`

## 7. Keys, Constraints, and Indexes
Minimum constraints:
- Primary key on `transactions_input.txn_id`
- Primary key on `accounts_input.account_id`
- Primary key on `customers_kyc_input.customer_id`
- Foreign key `accounts_input.customer_id` -> `customers_kyc_input.customer_id`
- Foreign key `transaction_channel_details.txn_id` -> `transactions_input.txn_id`
- Optional unique rule by rail for external references (for example `external_ref_type + external_ref_no + value_date`)

Recommended indexes:
- `idx_txn_from_time` on (`from_account`, `txn_timestamp` DESC)
- `idx_txn_to_time` on (`to_account`, `txn_timestamp` DESC)
- `idx_txn_customer_time` on (`customer_id`, `txn_timestamp` DESC)
- `idx_txn_channel_time` on (`channel`, `txn_timestamp` DESC)
- `idx_txn_status_time` on (`txn_status`, `txn_timestamp` DESC)
- `idx_txn_reference` on (`reference_txn_id`)
- `idx_txn_external_ref` on (`external_ref_type`, `external_ref_no`)
- `idx_account_status` on (`account_status`)
- `idx_customer_kyc_status` on (`kyc_status`)

## 8. Naming Compatibility Note
Current UniGRAPH ingest examples use `from_account` and `to_account`.
If your core SQL standard uses `from_account_id` and `to_account_id`, create compatibility views/aliases to avoid contract breakage.

## 9. Future Channels
If CARD, WALLET, NETBANKING, AEPS, or BBPS are added later:
- Keep `transactions_input` unchanged.
- Add only channel-specific optional fields in `transaction_channel_details`.
- Do not move fraud conclusions into source input tables.
